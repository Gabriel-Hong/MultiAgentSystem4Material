> AI를 단순히 "사용"하는 것이 아니라 "잘 활용"하기 위해 필요한 것들

| 순서 | 주제 | 트러블슈팅 | 청중 반응 유도 |

| 1 | **AI도 맥락이 필요하다** | 대용량 파일 압축 + 라인번호 포맷팅 | "17,000줄을 어떻게?" → "이렇게 했구나!" |
| 2 | **AI의 출력은 후처리가 필수** | JSON 파싱, 인코딩, 줄바꿈 | "AI가 완벽하지 않구나" → "이렇게 보완하는구나" |

### 각 파트별 발표 스크립트 가이드

#### Part 1: AI도 맥락이 필요하다

> "AI에게 17,000줄 코드를 주면 어떻게 될까요?"

**문제 제시:**
- GPT-4의 컨텍스트 윈도우 한계
- 토큰 비용 폭발
- 정확도 저하

**해결 보여주기:**
- Clang AST로 구조만 추출 → 97% 압축
- 라인번호 포맷팅 → 정확한 위치 파악

**교훈:**
> "AI에게는 전체가 아니라 핵심만 보여주세요"

---

#### Part 2: AI의 출력은 후처리가 필수

> "AI가 생성한 JSON이 파싱이 안 됩니다"

**문제 제시:**
- trailing comma, 제어문자 등으로 JSON 파싱 실패
- 인코딩 변환으로 한글 깨짐
- 줄바꿈 변환으로 전체 파일이 변경된 것처럼 보임

**해결 보여주기:**
- 정규식 후처리 → 파싱 성공률 75% → 98%
- chardet으로 원본 인코딩 감지 & 보존
- 원본 줄바꿈 스타일 유지

**교훈:**
> "AI의 출력을 그대로 쓰지 마세요. 검증하고 정제하세요"

---

#### Part 3: AI는 가드레일이 필요하다

> "AI가 잘못된 코드를 생성하면 어떻게 하죠?"

**문제 제시:**
- LLM 환각(Hallucination) 위험
- 범위 외 요청 처리 위험
- 자동화 시스템의 신뢰성 문제

**해결 보여주기:**
- 3단계 검증 시스템
  - Layer 1: 신뢰도 검증 (≥ 0.9)
  - Layer 2: Health Check
  - Layer 3: Timeout Guard (300s)

**교훈:**
> "자동화할수록 안전장치가 중요합니다"

---

#### Part 4: AI는 비용 관리가 중요하다

> "LLM API 비용이 너무 많이 나옵니다"

**문제 제시:**
- 동일 요청에 대한 중복 처리
- API Rate Limit
- 예측 불가능한 비용

**해결 보여주기:**
- Redis 캐싱 전략
  - Intent Classification: 24시간 캐싱
  - Bitbucket API: 5분 캐싱
  - LLM 응답: 24시간 캐싱

**교훈:**
> "캐싱은 비용과 성능, 두 마리 토끼를 잡습니다"

### Tier 1-2: 인코딩 보존

#### 문제

```
Bitbucket에서 파일을 가져와서 수정 후 커밋했더니...
- Git diff에서 전체 파일이 변경된 것으로 표시
- 한글 주석이 깨짐
- 코드 리뷰가 불가능
```

#### 원인

```python
# 문제의 코드
content = response.text  # Python이 자동으로 UTF-8로 변환
                         # 원본이 EUC-KR/CP949면 인코딩 정보 손실
```

#### 해결: chardet으로 원본 인코딩 감지 & 보존

```python
import chardet

def get_file_with_encoding(file_content_bytes):
    """원본 인코딩 감지 및 보존"""
    # 1. 바이너리로 파일 읽기
    raw_content = response.content  # .text 대신 .content 사용
    
    # 2. 원본 인코딩 감지
    detected = chardet.detect(raw_content)
    original_encoding = detected['encoding']  # 예: 'EUC-KR'
    
    # 3. 수정 작업 수행
    text_content = raw_content.decode(original_encoding)
    modified_content = modify_code(text_content)
    
    # 4. 원본 인코딩으로 다시 인코딩
    final_content = modified_content.encode(original_encoding)
    
    return final_content
```

#### 결과

| 지표 | Before | After |
|------|--------|-------|
| 한글 주석 보존 | ❌ 깨짐 | ✅ **100% 보존** |
| Git diff 정확도 | ❌ 전체 변경 | ✅ **실제 수정만** |
| 인코딩 감지 | 수동 | ✅ **자동** |

### Tier 2-9: 라인번호 포맷팅

> ⭐ Copilot, Claude Code 사용자에게 바로 적용 가능한 팁

#### 문제

```
LLM에게 "10732번 줄을 수정해줘"라고 해도...
- 정확한 위치 파악 실패
- 엉뚱한 곳을 수정
- 전체 파일을 주면 토큰 낭비 + 정확도 저하
```

#### 해결: 라인번호 포맷팅 + 관련 함수만 추출

```python
def format_code_with_line_numbers(code: str) -> str:
    """코드에 라인번호를 붙여서 LLM이 정확한 위치 파악 가능"""
    lines = code.split('\n')
    return '\n'.join(f"{i+1:5d}| {line}" for i, line in enumerate(lines))
```

**Before: 라인번호 없음**
```cpp
void GetSteelList() {
    // 어디가 어딘지 모름
    if (code == "SS400") {
        return data;
    }
}
```

**After: 라인번호 포함**
```cpp
10732| void GetSteelList() {
10733|     // 정확한 라인 위치 파악 가능
10734|     if (code == "SS400") {
10735|         return data;
10736|     }
10737| }
```

### Tier 2-5: LLM 응답 안정화

#### 문제

```
LLM이 JSON으로 응답하도록 했는데...
- trailing comma로 파싱 실패: [{...}, ]
- 제어문자(탭, 줄바꿈)가 문자열 안에 포함
- 마크다운 코드블록으로 감싸서 응답
```

#### 해결: 다단계 후처리

```python
def parse_llm_json_response(response: str) -> dict:
    """LLM JSON 응답 안정적으로 파싱"""
    
    # 1. 마크다운 코드블록 추출
    if "```json" in response:
        response = extract_json_from_markdown(response)
    
    # 2. Trailing comma 제거
    response = re.sub(r',(\s*[}\]])', r'\1', response)
    
    # 3. 제어문자 이스케이프
    response = escape_control_chars_in_strings(response)
    
    # 4. JSON 파싱
    return json.loads(response)
```
